В отличие от универсальных черновиков (инструкций), **кастомизации** — это платформо-зависимые файлы (скиллы, правила), собираемые (компилируемые) под конкретного менеджера агентов.

Эта инструкция описывает процесс сборки кастомизаций на основе файлов из папки [drafts](<../../ai_artifacts/drafts/>).

## 1. Определение сущностей и сбор контекста

**СТРОГОЕ ПРАВИЛО:** Прежде чем переходить к созданию структуры и файлов, агент **ОБЯЗАН** выполнить подготовительный этап работы с документацией. Категорически запрещено приступать к компиляции "вслепую", додумывать форматы или опираться только на базовые знания.

### 1.1. Целевые сущности
У нас есть универсальные черновики. Для их реализации в конкретных менеджерах определены следующие сущности:
- **Antigravity**: **Плагин** (Plugin).
- **Codex**: **Плагин** (Plugin).

### 1.2. Сбор технических требований (Исследование/Research)
Агент обязан провести строгое и сфокусированное исследование документации для сбора точного контекста. Запрещено использовать поверхностные предположения. При сборе информации критически важно отсекать избыточные данные: наполняйте контекст только **необходимыми и достаточными сведениями**, исключая любой информационный шум.

Алгоритм исследования:
1. Обратитесь к файлу [Ссылки](<../Ссылки.md>) за официальной документацией.
2. Выполните поиск информации и детально изучите документацию, чтобы понять внутреннюю структуру целевого инструмента.
3. Выделите только ключевые технические спецификации: необходимый состав конфигурационных файлов, манифесты, обязательные поля в них и иерархию директорий.
4. Отсейте лишние справочные тексты, примеры использования и нерелевантные инструкции, оставив в контексте исключительно сухие технические правила сборки.
5. Строго следуйте собранным техническим спецификациям на последующих этапах генерации.

Компиляция представляет собой направленное преобразование исходных данных и контекста в файлы кастомизаций:

### Преобразования для Antigravity (в директорию `ai_artifacts/integrated/antigravity/`)
| Входные данные (Источник) | Логика трансформации | Целевой файл кастомизации |
| :--- | :--- | :--- |
| Спецификации плагина из документации (Шаг 1.2) | Создание манифеста плагина согласно извлеченным требованиям к структуре | `plugin.json` |
| Черновик `drafts/{name}.md` с условием [Без явного вызова](<../Структура пака/main.md#без-явного-вызова>) | Трансляция содержимого черновика в файл правила. Добавляется YAML-заголовок:<br>`---`<br>`trigger: "model_decision"`<br>`description: "..."`<br>`---` | `rules/impl-instrs-{name}.md` |
| Черновик `drafts/{name}.md` с условием [По явному вызову](<../Структура пака/main.md#по-явному-вызову>) | Трансляция содержимого черновика в SKILL.md. Добавляется YAML-заголовок:<br>`---`<br>`name: "impl-instrs:{name}"`<br>`description: "..."`<br>`---` | `skills/{name}/SKILL.md` |

| :--- | :--- | :--- |

## 3. Трансляция условий в метаданные

При переводе абстрактных условий черновика (см. [Переиспользуемые условия](<../Структура пака/main.md#переиспользуемые-условия>)) в YAML-заголовки плагина, руководствуйтесь следующими инженерными принципами.

### Условие: «Реагирует на файлы инструкций»
* **Реализация (все менеджеры):** Вы обязаны встроить в поле `description` точные паттерны обнаружения, описанные в [концепте](</ai_instrs.concept/Состав пака/Работа с файловой структурой и рабочим пространством/main.md#обнаружение-инструкций>). 
* **Критические блокираторы:** Если черновик содержит критически важную логику (отменяет базовое поведение), используйте ультимативный тон. Это служит «крючком» для механизма внимания (attention) нейросети на этапе сканирования и гарантирует строгую проверку паттернов:
  ```yaml
  description: "КРИТИЧЕСКАЯ БЛОКИРОВКА: ВЫ ОБЯЗАНЫ прочитать это перед ЛЮБЫМ взаимодействием (даже при простом исправлении опечаток) с файлами и папками по паттернам: 'ai_instrs/', '*.ai_instrs/', 'ai_instrs.*/', 'ai_instrs.*', '*.ai_instrs.*'. Файл содержит нестандартные правила, которые отменяют ваши базовые знания по умолчанию. Вы провалите задачу, если проигнорируете это."
  ```

### Условие: «По явному вызову»
* **Реализация:** Для всех менеджеров (Antigravity и Codex) всегда компилируется как **скилл**.
* **Блокировка авто-активации:** Чтобы LLM-роутер не подтянул этот скилл в фоновом режиме, поле `description` должно начинаться с запрета:
  ```yaml
  description: "ВЫЗЫВАЕТСЯ ИСКЛЮЧИТЕЛЬНО ПОЛЬЗОВАТЕЛЕМ (ОПЕРАТОРОМ) ВРУЧНУЮ. Запрещено автоматическое фоновое срабатывание. [Краткая суть скилла для оператора]"
  ```
  > **Примечание:** Эта конструкция «обманывает» роутер, заставляя его пропустить скилл при фоновом поиске, но скилл безусловно загрузится при прямом вызове `/<имя-скилла>`.

### Условие: «Без явного вызова»
* **Antigravity:** Компилируется как **правило** (rule). Чтобы правило начало динамически реагировать на текст как скилл, добавьте параметр `trigger: model_decision`. Условия срабатывания опишите в `description`.
* **Codex:** Так как Codex не имеет сущности «правил», компилируется как **скилл**. Скрыть его из меню слэш-команд в Codex технически невозможно, поэтому просто напишите максимально точный `description` для строгой отработки роутером.
