# Мета-инструкции

**Суть:** Способность писать инструкции на любом уровне, в том числе инструкции для инструкций.

## Жёсткий императивный алгоритм работы

При получении задачи на создание или изменение мета-инструкции, неукоснительно выполняй следующие шаги в указанном порядке:

### Шаг 1. Анализ текущей базы
1. Обнаруживай и читай существующие правила, используя паттерны файлов и папок `ai_instrs`.
2. Определи, к какому уровню абстракции относится задача (например, базовое поведение агента, правила написания промптов, структура директорий).
3. Проверь концепцию на отсутствие пересечений с уже существующими инструкциями. Строго соблюдай принцип изолированных зон ответственности.

### Шаг 2. Трансляция концепций в алгоритмы
Запрещается переносить абстрактные бизнес-требования напрямую. Каждую концептуальную идею переводи в исполняемый алгоритм:
1. Замени абстрактные намерения (например, "агент должен проанализировать ситуацию") на конкретные шаги вызова инструментов (например, "Шаг 1: Используй инструмент X. Шаг 2: Выполни поиск Y").
2. Сформулируй единый, четкий и однозначный свод правил для целевого объекта.
3. Опиши триггеры активации, если инструкция должна срабатывать автоматически (в соответствии со стандартами `trigger: model_decision`).

### Шаг 3. Взаимосвязи и переиспользование
1. Если новая мета-инструкция использует логику, описанную в другой инструкции, используй явные ссылки в формате `[Название](<Путь>)`.
2. Никогда не дублируй логику из смежных объектов.

### Шаг 4. Форматирование и валидация
1. Оформи итоговый документ в формате Markdown с обязательным YAML-заголовком (если применимо).
2. Убедись, что итоговый текст не содержит двояких толкований, а представляет собой строгий технический пайплайн.

### Шаг 5. Применение
1. Сформируй имя файла, строго соответствующее паттернам обнаружения (например, `название.ai_instrs.md`).
2. Сохрани файл в рабочей директории, используя исключительно разрешенные относительные пути.
