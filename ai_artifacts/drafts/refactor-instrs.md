## Содержание
# Навык: Рефакторинг инструкций

Этот навык предоставляет жёсткий императивный алгоритм для реорганизации и улучшения существующих файлов инструкций. Его главная цель — сохранить семантику исходных инструкций, улучшив при этом структуру файлов и обеспечив строгую изоляцию контекста для более качественного выполнения задач агентами.

## 📋 Жёсткий алгоритм выполнения

При вызове данного скилла оператором, строго следуй пошаговому алгоритму.

### Шаг 1: Анализ исходных данных
1. Вызови `view_file` для чтения целевых файлов инструкций, указанных оператором. Если файлы не указаны, используй `list_dir` или `grep_search` для поиска файлов по паттернам `ai_instrs`.
2. Извлеки и зафиксируй исходную семантику, требования и логику из инструкций. Ни одно бизнес-требование не должно быть утеряно при рефакторинге.

### Шаг 2: Оценка изоляции контекста
1. Проанализируй каждую инструкцию на предмет наличия конфликтных или избыточно подробных этапов (например, объединение глубокого поиска информации и написания сложного кода в одном файле).
2. **Правило разделения:** Если подробное описание одного шага мешает фокусу на другом, прими решение об их разделении на независимые инструкции.
3. Спроектируй сценарий, при котором оператор сможет давать агенту первую инструкцию, а после её полного выполнения в изолированном контексте — вторую.

### Шаг 3: Проектирование файловой структуры
1. Спланируй новую иерархию файлов и папок.
2. **Ограничения путей:** При упоминании путей к файлам и папкам используй **только относительные пути** (от рабочей директории или от файлов внутри неё). Использование абсолютных путей строго запрещено.
3. Убедись, что новые имена соответствуют паттернам обнаружения инструкций (например, папка `ai_instrs/` или суффикс файла `.ai_instrs.md`).
4. **Артефакты:** Если в процессе рефакторинга тебе нужно создать план, отчет или скрипт, сохрани его в директории `./ai_artifacts/` (относительно файла инструкции или рабочей директории).

### Шаг 4: Применение изменений
1. Используй `write_to_file` для создания новых файлов инструкций и `replace_file_content` (или `multi_replace_file_content`) для обновления существующих.
2. При создании ссылок между инструкциями используй формат markdown со строгим соблюдением угловых скобок для путей: `[Название шага](<относительный_путь_к_файлу>)`.
3. Убедись, что файлы физически разделены согласно плану из Шага 2, предотвращая загрязнение контекста.

### Шаг 5: Верификация
1. Сравни результат с исходной семантикой, зафиксированной на Шаге 1.
2. Убедись, что смысл инструкций сохранен на 100%, а изменения коснулись только структуры и распределения фокуса.
3. Выведи для оператора отчет об изменениях и предоставь инструкции (или ссылки на них) для дальнейшего использования.
